One deployable unit

The monolith: everything ships together

  • Single codebase, single deployment artifact (WAR/JAR)
  • All modules share one database
  • Works well for small teams and early-stage products
  • Scaling means scaling everything, even unused parts

The Monolith Problem05 / 21

What microservices solve — and what they don't

Independent deployment
Each service ships on its own schedule — no coordinating a big-bang release across the whole system.
Independent scaling
Scale only the services under load, not the entire application. Pay for what you actually use.
Team autonomy
Small teams own their service end-to-end — tech choices, database, release cadence. No waiting on others.
Real cost: complexity
Networking, failures, and distributed state replace simple method calls. This is not free.
⚠ Know the tradeoff

Service Boundaries

How do you know where to cut?

Service Boundaries07 / 21
Domain-Driven Design

Bounded contexts: the unit of a microservice

  • Term from Domain-Driven Design (DDD)
  • A logical boundary around a domain concept with its own language, data model, and rules
  • Services map 1:1 to bounded contexts in most cases
  • The boundary says: this service does not need to know what happens over there
Heuristic

A service should be ownable by a team of 2–8 people.

The 12-Factor App

A methodology for building robust, portable services

The 12-Factor App09 / 21
A contract for cloud-ready services

Authored at Heroku, 2011

  • Written by Adam Wiggins — 12 principles for software-as-a-service
  • Language- and platform-agnostic, maps naturally to Spring Boot
  • Designed to: maximize portability, enable cloud deployment, support continuous delivery
  • A service can follow 8 of 12 and still be far more cloud-ready than a typical monolith

Know the rule before you break it. Teams violate factors intentionally — with good reason.

12
Independently applicable factors
Each one is a guideline, not a law

The 12-Factor App10 / 21

Factors 1–6: Codebase to processes

I
Codebase
One repo, many deploys — no separate codebases per environment
II
Dependencies
Explicitly declare all dependencies (Maven/Gradle pom.xml) — never rely on system-installed libraries
III
Config OFTEN VIOLATED
Store config in environment variables, not code — URLs, credentials, feature flags
IV
Backing Services
Treat databases, queues, and caches as attached resources — swappable without code changes
V
Build / Release / Run
Strictly separate build, release, and runtime stages — never patch code in production
VI
Processes OFTEN VIOLATED
Services are stateless — any state lives in a backing service (DB, Redis), not in memory
Entry-Level Java Engineering TrackModule 1
The 12-Factor App11 / 21

Factors 7–12: Port binding to logs

VII
Port Binding
Services expose themselves via a port — Spring Boot's embedded Tomcat is the canonical example
VIII
Concurrency
Scale out by running more instances (horizontal), not bigger instances (vertical)
IX
Disposability CRITICAL IN K8S
Fast startup, graceful shutdown — services must handle being killed and restarted at any time
X
Dev/Prod Parity
Keep development, staging, and production as similar as possible
XI
Logs SURPRISES STUDENTS
Treat logs as event streams — write to stdout, let the platform aggregate (no log files)
XII
Admin Processes
Run one-off tasks (DB migrations, scripts) as separate processes, not baked into the service

Cloud-Native Microservices

Beyond 12-factor: what makes a service truly cloud-ready

Cloud-Native Microservices14 / 21

Six hallmarks of a cloud-native service

01
Containerized
Packaged as a Docker image — same artifact runs in dev, staging, and prod
02
Dynamically orchestrated
Kubernetes schedules, scales, and heals it — the service doesn't know which host it's on
03
Health-aware
Exposes /actuator/health — the platform uses it to route traffic and restart failed instances
04
Observable
Structured logs, metrics (Micrometer), and traces — not just "it worked or it didn't"
05
Resilient by design
Assumes dependencies fail — uses timeouts, retries, and circuit breakers
06
Independently deployable
No coordinated release trains — deploys whenever its tests pass

Cloud-Native Microservices15 / 21
Why Spring Boot fits

Built for the cloud-native model

  • Embedded server (Tomcat/Netty) — no external app server required (Factor VII)
  • Spring Boot Actuator — /health, /metrics, /info out of the box
  • Spring Profiles — environment-specific config without code changes (Factor X)
  • Graceful shutdown built-in — safe Kubernetes termination (Factor IX)
  • Micrometer included — metrics to Prometheus, Datadog, or CloudWatch
  • Docker-friendly — mvn spring-boot:build-image builds an OCI image
application.yml
# Factor III: config in environment spring: datasource: url: ${DB_URL} # injected at runtime username: ${DB_USER} password: ${DB_PASSWORD} server: port: 8080 shutdown: graceful # Factor IX management: endpoints: web: exposure: include: health,info,metrics

Live demo: start a bare Spring Boot app, hit /actuator/health — students are always surprised how much comes for free.

Spring Boot Actuator

Production-ready endpoints — zero configuration required

Spring Boot Actuator17 / 21
What you get for free

Add one dependency. Get production visibility.

pom.xml
<dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-actuator</artifactId> </dependency>
application.yml — expose endpoints
management: endpoints: web: exposure: include: health,info,metrics,env endpoint: health: show-details: always
Key endpoints
/actuator/health
Service status + component health (DB, disk, custom). Used by Kubernetes liveness & readiness probes.
/actuator/metrics
JVM heap, GC, HTTP request counts, latency percentiles. Scraped by Prometheus in Module 4.
/actuator/info
Custom metadata: app version, build time, git commit. Populated from application.yml or build plugins.
/actuator/env
Active profiles and resolved config properties. Verify Factor III (config in environment) is working.
/actuator/loggers
View and change log levels at runtime — no restart needed. Invaluable in production debugging.

Spring Boot Actuator18 / 21
Kubernetes integration

Health probes & graceful shutdown

Kubernetes needs to know when your service is ready to receive traffic and when it is alive. Actuator provides both endpoints out of the box.

deployment.yaml — probe config
livenessProbe: httpGet: path: /actuator/health/liveness port: 8080 initialDelaySeconds: 30 periodSeconds: 10 readinessProbe: httpGet: path: /actuator/health/readiness port: 8080 initialDelaySeconds: 15 periodSeconds: 5
Factor IX — Disposability

Graceful shutdown: finish what you started

When Kubernetes sends SIGTERM, Spring Boot stops accepting new requests and waits for in-flight requests to complete before shutting down.

application.yml
server: shutdown: graceful spring: lifecycle: timeout-per-shutdown-phase: 30s # Spring Boot then: # 1. Sets readiness to DOWN # (Kubernetes stops sending traffic) # 2. Waits up to 30s for in-flight requests # 3. Shuts down cleanly

Without graceful shutdown, a pod restart during checkout drops the user's order mid-transaction.

Pitfalls engineers hit in their first microservice project

MistakeBetter Approach
Config in code: new URL("http://localhost:8082")Environment variable: ${ORDER_SERVICE_URL}
Shared database between servicesEach service owns its schema — communicate via API or events
Session state in memoryExternalize to Redis — services must be stateless
Logging to files on diskLog to stdout — let the platform collect and aggregate
One big release every two weeksEach service deploys independently when its pipeline passes
Splitting too early ("nano-services")Start coarser; split only when a team feels the friction
