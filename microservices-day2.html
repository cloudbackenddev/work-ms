How microservices fail — and why it's subtle

Scenario A — Slow dependency
Not dead, just slow
Service B takes 30s to respond. A's threads pile up waiting. A runs out of threads. A stops responding to its own callers. C, which calls A, also freezes. B caused it — but A looked like the failure.
Scenario B — Partial failure
Accepts requests, fails at the DB
Payment service is up, but its DB connection pool is exhausted. It accepts requests, processes them, then fails at the write. Callers see a 500. The service looks healthy from the outside.
Scenario C — Network partition
Two instances, inconsistent data
Two inventory instances can talk to clients but not each other. Both accept stock updates. When the partition heals, data is inconsistent. Neither instance knew it was wrong.
Scenario D — Clock drift
400ms apart — log correlation breaks
Two services timestamp events using their system clock. The clocks are 400ms apart. "A happened before B" is no longer reliable. Debugging becomes archaeology.

Resilience Patterns

Timeouts · Retries · Circuit Breakers

Resilience Patterns07 / 22

Three tools every service needs

01 — Timeout
Set a deadline on every call
Every outbound call must have a deadline. If the dependency doesn't respond in N ms, give up and handle the failure. Without a timeout, one slow dependency exhausts all your threads.
// WebClient .timeout(Duration.ofMillis(500))
02 — Retry
Transient failures often self-heal
Brief network blips and restarting pods often resolve on the next attempt. Retry automatically — but with limits: max attempts, exponential backoff, and jitter to avoid thundering herd.
@Retry(name = "payment-service") // configure in application.yml
03 — Circuit Breaker
Stop calling what's broken
If a dependency is consistently failing, stop calling it. Open the circuit, return a fallback immediately, and probe periodically to check recovery. Protects threads, prevents cascade.
@CircuitBreaker(name = "payment-service", fallbackMethod = "paymentFallback")

How a circuit breaker actually works

  • CLOSED: All calls go through. Failures are counted.
  • OPEN: All calls fail immediately with fallback. No requests sent.
  • HALF-OPEN: A few probe calls test recovery.
Spring Boot config
resilience4j: circuitbreaker: instances: payment-service: slidingWindowSize: 10 failureRateThreshold: 50 waitDurationInOpenState: 30s permittedNumberOfCallsInHalfOpenState: 3
Key insight

OPEN state means your service is protecting itself by failing fast — keeping threads free for requests that CAN succeed.

Resilience Patterns09 / 22
Resilience4j in practice

Circuit breaker in code

Step 1 — pom.xml
<!-- resilience4j --> <dependency> <groupId>io.github.resilience4j</groupId> <artifactId>resilience4j-spring-boot3</artifactId> </dependency> <!-- required for AOP proxying --> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-aop</artifactId> </dependency>
Step 3 — fallback contract

Don't just throw in the fallback. Return something meaningful: a default value, a cached result, a "pending" status, or queue the work for retry. The fallback is your contract with the caller when the dependency is unavailable.

Step 2 — annotate the call
@Service public class OrderService { @CircuitBreaker( name = "payment-service", fallbackMethod = "paymentFallback") @Retry(name = "payment-service") public PaymentResponse chargeCustomer( OrderRequest order) { // remote call return paymentClient.charge(order); } // Called when OPEN or retries exhausted private PaymentResponse paymentFallback( OrderRequest order, Exception ex) { log.warn("Payment unavailable: {}", ex.getMessage()); return PaymentResponse .queued(order.getId()); } }

Resilience Patterns10 / 22
Bonus pattern 1

Bulkhead

Isolates thread pools per dependency. Even if payment-service calls pile up, they consume threads from a separate pool — order-service's own thread pool stays free.

Think of ship bulkheads: one compartment floods, the others stay dry.

When to use

When one slow dependency must not starve all other operations.

@Bulkhead(name = "payment-service", type = THREADPOOL)
Bonus pattern 2

Rate Limiter

Caps how many calls your service makes to a downstream in a given time window. Prevents overwhelming a dependency that has its own limits.

Also used at the API Gateway to protect your own services from abusive clients.

When to use

Calling third-party APIs with rate limits (Stripe, Twilio) or protecting a slow internal service.

@RateLimiter(name = "payment-service")

Distributed Data

Why you can't use transactions across services — and what to do instead

Distributed Data12 / 22
The rule

Each service owns its data. Full stop.

  • Each microservice has its own database or schema
  • No other service queries it directly — only via the service's API
  • Enforces the service boundary at the data layer
  • Enables independent schema evolution — order-service can add a column without touching user-service
  • Enables polyglot persistence — each service picks the right DB for its needs

This is the rule most teams break first — usually out of convenience ("I'll just do a JOIN"). The short-term gain is a long-term coupling that prevents independent deployment.

Distributed Data13 / 22
The core problem

Why ACID transactions don't work across services

A customer places an order. Three things must happen atomically:

  • Reserve stock in inventory-service
  • Charge the customer in payment-service
  • Create the order record in order-service
The problem

If payment succeeds but inventory reservation fails, you have charged the customer for something you can't fulfil. There is no rollback across services.

In a monolith
BEGIN TRANSACTION INSERT INTO orders ... UPDATE inventory SET stock = stock - 1 ... INSERT INTO payments ... COMMIT -- all or nothing
In microservices
// No shared transaction. // Each service has its own DB. // 2PC exists but is fragile, // slow, and avoided in practice. // Instead: design for // eventual consistency

Distributed Data14 / 22

Sagas: consistency without a transaction

A sequence of local transactions. Each step publishes an event. If a step fails, compensating transactions undo previous steps.

Choreography — event-driven
order-service → publishes: OrderCreated inventory-service → reserves stock → publishes: StockReserved payment-service → charges card → publishes: PaymentCharged order-service → confirms order On failure: payment-service → publishes: PaymentFailed inventory-service → releases stock order-service → cancels order

Services don't know each other. Hard to see overall flow. Best for simple, linear flows.

Orchestration — central coordinator
SagaOrchestrator: Step 1 → inventory.reserve() ↳ fail → STOP Step 2 → payment.charge() ↳ fail → inventory.release() → STOP Step 3 → order.confirm() ↳ fail → payment.refund() + inventory.release()

Central place to observe and debug. Best for complex flows with many branches.

Distributed Data15 / 22
The unsung hero

Every mutation must be safe to call twice

Retries are everywhere — client retries, Resilience4j, message queue redeliveries. If your endpoint is not idempotent, a retry creates a double charge or duplicate order.

  • Client generates a unique idempotency-key (UUID) per operation
  • Server stores (key → result) in a database
  • On duplicate request: return the stored result, don't re-execute

Stripe has done this since day one. Every mutation endpoint requires an Idempotency-Key. It's why Stripe can safely retry failed payment requests without double-charging.

Spring Boot implementation
@PostMapping("/payments") public ResponseEntity<PaymentResponse> charge( @RequestHeader("Idempotency-Key") String idempotencyKey, @RequestBody PaymentRequest request) { // Check if we've seen this key Optional<PaymentResponse> existing = idempotencyStore.find(idempotencyKey); if (existing.isPresent()) { // return cached result return ResponseEntity.ok(existing.get()); } PaymentResponse result = paymentProcessor.charge(request); idempotencyStore.save(idempotencyKey, result); return ResponseEntity.ok(result); }

Observability

Logs · Metrics · Distributed Tracing

Observability17 / 22

Knowing what your system is doing without being there

A single user request may touch 8 services across 20 pods. A 500 error could originate anywhere. Without observability, you have no way to find it.

Pillar 1 — Logs
What happened
A timestamped record of events. Use structured logging (JSON). Always include traceId and spanId so logs from different services can be correlated.
SLF4J → Logback (JSON) → stdout → ELK / Loki / CloudWatch
Pillar 2 — Metrics
How it's performing
Numeric measurements over time: request rate, error rate, latency percentiles, JVM heap, DB pool size. Aggregated and visualised on dashboards.
Micrometer (Actuator) → Prometheus → Grafana
Pillar 3 — Distributed Traces
Where the time went
A trace records the full journey of one request across multiple services. A span is one unit of work. Traces answer: "Which service caused the 3-second latency?"
Micrometer Tracing (Spring Boot 3) → Zipkin / Jaeger

Observability18 / 22
Structured logging

Logs that machines can read and correlate

  • Plain text logs are for humans in one service
  • JSON logs can be queried and aggregated across all services
  • Always include: timestamp, level, service, traceId, spanId, message
  • Never log sensitive data (passwords, card numbers, PII)
logback-spring.xml
<!-- logback-spring.xml --> <encoder class="net.logstash.logback .encoder.LogstashEncoder" />

The traceId is the magic field. Paste it into your log aggregator and see every log line from every service involved in one user's request.

The CAP Theorem

The fundamental tradeoff of distributed systems

CAP Theorem21 / 22
Brewer's theorem

You can have two of three. Choose wisely.

C
Consistency
Every read returns the most recent write. All nodes see the same data.
A
Availability
Every request receives a response. The system is always up.
P
Partition Tolerance
System keeps operating when network messages between nodes are lost. Not optional — networks partition.

The real choice is CP or AP. During a partition: return stale data (choose A), or refuse to respond (choose C)?

Examples & microservice impact
CP SystemsAP Systems
HBase, ZooKeeperCassandra, DynamoDB
Refuse requests during partitionReturn stale data during partition
Banking, financial ledgersShopping carts, social feeds, DNS
In your microservices

When inventory-service is partitioned from order-service:

  • CP choice: order-service refuses to create orders (safe, unavailable)
  • AP choice: order-service creates the order using last-known inventory (available, possibly oversells)

Both are valid. The business decides which failure mode is acceptable.

What to carry forward from Module 4

Key Takeaways

01
The network will fail
Every remote call needs a timeout, retry, and circuit breaker. This is not optional.
02
Cascading failures are the silent killer
One slow dependency can take down your entire service mesh. Circuit breakers stop the cascade.
03
No distributed transactions
Design for eventual consistency from day one. The Saga pattern gives you a structured way to manage multi-step operations.
04
Idempotency is mandatory
Any endpoint that mutates state must be safe to call more than once. Retries are everywhere.
05
Observability is infrastructure
Structured logs, metrics, and distributed traces are the minimum to operate in production.
06
The traceId is your best debugging tool
One ID shows you everything that happened across every service for one user request.
07
CAP is a business decision
Consistency vs. availability during a partition is a product requirement. Involve your team.
08
Complexity is the cost
Everything in this module is complexity a monolith gave you for free. Make sure you're at the scale that justifies it.
